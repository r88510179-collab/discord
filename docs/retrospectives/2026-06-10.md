## Evening ship — v606

Two grader correctness fixes merged and deployed together as **v606** (built
from clean `main`, `fly deploy --no-cache`, ~18:40Z):

### #73 (`31fd814`) — grade COMPLETE 1-leg parlays
A single pick stored as a 1-leg parlay (one `parlay_legs` row) was permanently
ungradeable: the ≤1-leg dispatch guard rejected every parlay with ≤1 recorded
leg. New pure helper `parlayLegDataComplete(description, legCount)` distinguishes
a **complete** 1-leg parlay (description names exactly one pick — one `•` — and
one leg exists) from genuine missing legs. The guard now skips only when *not*
complete; the missing-legs / 0-leg rejection and its exact reason string are
byte-identical, and multi-leg (≥2) parlays are untouched. 14 unit tests.

### #74 (`4c992c9`) — search backend honesty (M-3) + query ordinal fix
Every backend (Bing/Brave/DDG/Serper) now routes parsed results through
`assessSearchResults` before recording success: `parse_empty` (0 usable hits)
registers a circuit failure for *every* backend; Bing-only `generic_news`
(parsed but no hit mentions a query token) falls through *without* tripping the
breaker. `recordBackendResult` stamps `lastSuccess` only on a real success, so
`getBackendSnapshot` / `/admin` stop scoring drifted 200s as healthy. Bing/Serper
stay un-gated → Bing-first chain order preserved. Part 2: `extractSubject`
sentinel-protects ordinal/period qualifiers (`1st`–`4th`, `1H`/`2H`, `1Q`–`4Q`,
`F5`) so `"1st Quarter"` survives the numeric strip. 48 unit tests + patched
`search-chain-order` (8).

## Pool reset — decision & rationale

**298** bets in `grading_state='backoff'` had `grading_attempts` reset to **0**
(read-only-guarded write; pool-size guard 250–350; `grading_next_attempt_at`
left untouched). Rationale: those counters accrued during the **broken-search
era** — 264 of the 298 were at ≥5 attempts and would auto-void after just 1–2
*honest* tries. Resetting lets the now-honest search pipeline (S2/#74) judge each
bet on real source availability rather than inheriting burned counters. This is
also a precondition for **S1 measurement**, which should run after ~1 week of
honest attempts so the by-source cut is trustworthy.

Two write attempts during the session **aborted by design before touching a
row** (wrong column name; an 8-char-truncated bet id matching 0 rows). They are
written up as worked examples in `docs/RUNBOOKS/db-interventions.md` — the
origin of the two hard rules there (never trust a truncated id; never assume a
column name).

  path will now exhaust *honestly* (1–2 tries → no data → eventual void) instead
  of churning forever. Expect a wave of legitimate `auto_void_no_searchable_data`
  / `GRADE_BACKOFF_EXHAUSTED` as the backlog clears — this is intended (S5), not
  a regression. Note the **non-uniform auto-void rule** (BACKLOG): the two void
  paths key on different signals, so the void wave won't be a clean function of
  attempt count.
